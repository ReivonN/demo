When to use GET and POST?

GET, POST, PUT, HEAD, DELETE, PATCH, OPTIONS, CONNECT and TRACE are all HTTP methods with GET and POST being the most commonly used. These are the same methods that were previously mentioned in the ‘Request’ page as part of the request line.

Key Points on GET and POST requests

*according to w3schools.com*

  • GET requests can be cached
  • GET requests remain in the browser history
  • GET requests can be bookmarked
  • GET requests should never be used when dealing with sensitive data
  • GET requests have length restrictions
  • GET requests are only used to request data (not modify)
  • POST requests are never cached
  • POST requests do not remain in the browser history
  • POST requests cannot be bookmarked
  • POST requests have no restrictions on data length

We use GET to request and receive an asset from within the server and we use POST to create or update an asset in the server. However, GET is still capable of sending data but it is a lot riskier and less secure because the sent data becomes a part of a URL which can then be bookmarked and saved. It is highly recommended to use GET for its intended purpose of retrieving and to never use it when dealing with passwords or other sensitive information. POST offers a bit more security than GET because the parameters are not stored in the browser history or in the web server logs.

The general rule to follow when using these methods is to use GET when you want to retrieve data and use POST when you want to send data.
